# T-006 gRPC native IPC spike evidence

This workspace is a feasibility harness only. It is not a Cargo member of the
product workspace, is not imported by the hosted application, and is not
packaged or published.

## Pinned stack and topology

The browser uses the official `grpc-web` 2.0.2 binary runtime and generated
client. The Bun 1.3.13 terminator translates binary gRPC-Web frames to
`@grpc/grpc-js` 1.14.4 unary calls. Tonic 0.14.6 serves the native HTTP/2 gRPC
service over only:

- `unix:/absolute/path.sock` backed by a Tokio UDS on Linux GNU x64; or
- `unix:\\.\pipe\name` backed by a Tokio byte-mode named pipe on Windows MSVC
  x64.

There is no native TCP listener or alternate transport. The one TCP listener is
the Bun browser endpoint on `127.0.0.1`.

The remaining locked protocol/build versions are `google-protobuf` 3.21.4,
Prost 0.14.4, Tokio 1.53.0, `tokio-stream` 0.1.18,
`tonic-prost`/`tonic-prost-build` 0.14.6, and `protoc-bin-vendored` 3.2.0
(protoc 31.1). The committed browser client was generated by protoc 3.20.3,
whose built-in JavaScript generator is required by `protoc-gen-grpc-web`
2.0.2:

```sh
protoc -I proto \
  --plugin=protoc-gen-grpc-web=/path/to/protoc-gen-grpc-web-2.0.2 \
  --js_out=import_style=commonjs,binary:generated \
  --grpc-web_out=import_style=commonjs+dts,mode=grpcweb:generated \
  proto/spike.proto
```

Regenerating with those versions produces the committed files byte for byte.
The Bun server bundle uses CommonJS output so the unmodified official generated
CommonJS protobuf module retains its runtime semantics.

## Blocked dependency script

`bun ci --frozen-lockfile` reports only the transitive
`protobufjs@7.6.5` postinstall as blocked. Its installed
`scripts/postinstall.js` reads package manifests solely to warn about a
dependency version-prefix convention; it does not generate or mutate runtime
files. The harness does not trust or execute it, and the built Linux chain below
passes with it blocked.

## Native evidence status

On Linux x86_64, the following built-form probe passed locally on 2026-07-18
with Bun 1.3.13, Rust 1.88.0, and headless Chrome:

```sh
cargo +1.88.0 build --workspace --locked
bun run typecheck
bun run build
cargo +1.88.0 fmt --all -- --check
cargo +1.88.0 clippy --workspace --all-targets --locked -- -D warnings
cargo +1.88.0 test --workspace --locked
CHROME_BIN=/path/to/google-chrome \
  bun test --seed 20260718 \
  ./test/deadline-status.probe.ts ./test/interoperability.probe.ts
```

The probe passed success protobuf payloads, native initial metadata and OK
trailers, failed-precondition status plus custom trailers, deadline propagation
and Rust cancellation, browser abort and Rust cancellation, eight concurrent
calls, native unavailability without fallback, reconnect after native restart,
orderly shutdown, UDS removal, and browser-listener cleanup. `ss` showed the Bun
loopback listener and the UDS while showing no TCP listener owned by Rust.

The same test is wired to the standard GitHub-hosted `windows-2025` x64 runner.
That job builds before execution and uses the exact `@grpc/grpc-js` `unix:`
target against the Tokio byte-stream named pipe; it does not substitute a TCP
or other transport. Native Windows execution has not occurred in this local
worktree. Until that hosted job passes and its logs show the named-pipe marker,
no production library/topology selection is proven and T-007 or later gateway
work must not proceed.
